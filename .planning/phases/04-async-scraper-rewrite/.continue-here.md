## The 3 plans to write
**04-01 — Async fetch core** (Wave 1; SCRAPE-01/02/03 mechanism). New `fc26/ingest/web_async.py`:
- `AsyncFetcher` (shared `httpx.AsyncClient`, `Limits(max_connections=8, max_keepalive_connections=8, keepalive_expiry=30)`, `Timeout(connect=10, read=15, write=10, pool=5)`, UA `footie-playbook/0.1 (personal squad tool)`), `asyncio.Semaphore(concurrency)` global cap, `HostRateLimiter(min_interval)` per-host (netloc) min-interval + `random.uniform(0,min)` jitter. `concurrency`/`min_interval` are PARAMS (defaults 4 / 1.0). `fetch(url)`: per-host wait → semaphore → `client.get` → `raise_for_status` → **1 retry on ANY `httpx.HTTPError`** (NOT transport retries=, which only covers ConnectError) → raise `FetchError(f"could not fetch {url}: {last_error}")` (wording identical to `web.py:26`).
- `tests/test_web_async.py`: per-host interval enforced, jitter present, semaphore cap respected, retry-on-any-HTTPError (e.g. 500 retried once then FetchError), FetchError message identical.

**04-02 — Async ingest variants ALONGSIDE the sync ones** (Wave 2, dep 04-01; SCRAPE-03/04). Keep sync `enrich_cards`/`expand_cards`/`upgrade_card_images`/`refresh_data` (existing tests stay green); ADD async variants:
- `enrich_cards_async`: SERIAL skip-logic + URL resolution INCLUDING club discovery (builds `club_urls`/`club_pages` single-threaded — risk #3) → `asyncio.gather` the player-page fetches (error-isolated: each task returns (card, html|None, exc); never raises out) → SERIAL upsert loop in CARD order with `attempts`/`failures`/`ABORT_FAILURE_RATIO` check + `on_progress` (emit from serial consumer only — risk #4). Preserves EnrichResult tuples + merge order.
- `upgrade_card_images_async`: gather fetches → serial upsert in CARD order (matches the deterministic `workers=1` output, NOT the as_completed threaded order).
- `expand_cards_async`: keep SEQUENTIAL (pagination is data-dependent: `while True` breaks on empty/short page; `_resolve` uses `find_by_id` on prior upserts — risk #2 HIGH). Just route each page fetch through the AsyncFetcher (connection reuse). Ordering trivially preserved.
- `refresh_data_async(repo, *, min_ovr, concurrency, min_interval, on_progress, enrich_limit, manifest_path)`: `async with AsyncFetcher(...) as f:` wrap `with repo.batch():` → `await expand_cards_async(...)` then `await enrich_cards_async(...)`; manifest write unchanged.
- `tests/test_ingest_async.py`: FIXTURED equivalence — run sync vs async over the SAME fixtures + tmp repo; assert EnrichResult/ExpandResult/ImagesResult tuples EQUAL + `players.json` bytes IDENTICAL + on_progress sequence EQUAL. This is the byte-identical gate for the rewrite. (Reuse `tests/fixtures/*.html` + the injection seam.)

## Equivalence risks (ASYNC-SCRAPING §, MUST guard — guarded by test_ingest_async + golden)
1. Concurrent upsert corrupts data (CRITICAL) → exactly one serial writer; gather fetch/parse only.
2. `expand._resolve` id-suffix depends on upsert order (HIGH) → keep expand sequential.
3. enrich club cache shared dict (MED) → resolve all URLs (incl club discovery) before gather.
4. progress/log order (MED) → emit on_progress only from the serial consumer, never a gathered task.
5. merge_cards field precedence under reorder (MED) → preserve iteration/card order in serial upsert.
6. retry semantics (LOW/MED) → 1 retry on ANY httpx.HTTPError; identical FetchError wording.
