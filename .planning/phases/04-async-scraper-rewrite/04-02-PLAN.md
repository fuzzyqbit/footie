<threat_model>
Concurrency introduces the milestone's worst hazard: concurrent `repo.upsert` would corrupt `data/players.json` (whole-file read-modify-write → lost update). Mitigation is structural and mandatory: gather only fetch+parse; upsert SERIALLY on the consuming coroutine, in card/page order (mirrors the proven images.py workers>1 single-writer rule). Shared `club_urls`/`club_pages` cache is built single-threaded before gather. `on_progress` fires only from the serial consumer (deterministic order). Politeness is inherited from the AsyncFetcher (Plan 04-01).
</threat_model>

Structure (preserves order exactly):
1. **Serial pre-pass (single-threaded), in `repo.find_all()` order** — reproduce the sync loop's URL resolution WITHOUT fetching player pages concurrently yet:
   - Fetch the top-100 page once: `url_map = extract_player_urls(await fetcher.fetch(TOP100_URL))`.
   - Iterate cards in `repo.find_all()` order applying the SAME skip rules (futgg → skip; non-base → skip; not refresh and is_enriched → skip; `limit` cap on `attempts`). For each card that needs enriching, resolve its `player_url`: first `url_map.get(slugify(card.player_name))`; else fall back to club discovery via an async analog of `_discover_via_club` that builds `club_urls`/`club_pages` lazily but **serially in this pre-pass** (so the shared dicts are never mutated concurrently — risk #3). Misses recorded here go into `missed` in card order, identical wording.
   - Produce an ordered list `todo = [(card, player_url), ...]` plus the running `skipped`/`missed` exactly as the sync path would, and the `attempts` count (only resolved cards that will be fetched count toward `attempts`/limit, matching sync `enrich.py:69,81`).
2. **Concurrent fetch+parse, error-isolated** — `results = await asyncio.gather(*(_work(card, url) for card, url in todo))` where `_work` returns `(card, fresh_card|None, exc|None)` and NEVER raises (catch `FC26Error`). `_work` does `html = await fetcher.fetch(url)` then `parse_player_page(html, source_url=url)` (selectolax stays inline — small per-page parse). gather preserves input (card) order.
3. **Serial upsert in card order (single writer)** — iterate `results`; for each: if `exc`, `failures += 1; missed.append(f"{card.id}: {exc}")`; else `merged = repo.upsert(fresh)`; if `merged.id != card.id` emit the id-mismatch WARNING **verbatim** (byte-identical to enrich.py:88-91): `on_progress(f"WARNING: page id {merged.id!r} != card id {card.id!r} - original card left unenriched")`; then `enriched.append(merged.id)` and `on_progress(f"enriched {merged.id} ({merged.nation}, {merged.league})")`. Apply the abort check `if attempts >= ABORT_CHECK_AFTER and failures / attempts > ABORT_FAILURE_RATIO: raise ParseError(f"{failures}/{attempts} player pages failed - fcratings layout changed?")` — counting `attempts` over the resolved set (decide: increment in the serial loop per fetched card so the ratio matches the sync stream as closely as possible; document the chosen semantics in the summary). Return `EnrichResult(tuple(enriched), tuple(skipped), tuple(missed))`.

CRITICAL: `on_progress` is called ONLY in steps 1 (misses) and 3 (serial consumer) — never inside `_work`. The miss/enriched/skipped ordering must match the sync path on the same input; the equivalence test (Task 5) is the gate.
